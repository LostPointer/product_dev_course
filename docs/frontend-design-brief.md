# Experiment Tracking Portal — Design Brief

Документ для дизайнера. Описывает текущее состояние SPA-интерфейса
и всё необходимое для редизайна.

---

## 1. Что это за продукт

**Experiment Tracking Platform** — внутренний инструмент для исследователей и инженеров,
которые проводят и отслеживают эксперименты с физическими IoT-устройствами (RC-машинки,
датчики) и ML-пайплайнами. Аналог MLflow.

Основные задачи пользователя:
- Организовать работу по проектам и экспериментам
- Запустить Run (прогон эксперимента), отслеживать его статус
- Собрать телеметрию с физических датчиков в реальном времени
- Проанализировать метрики и данные после эксперимента
- Получать уведомления о событиях через Webhooks

Язык интерфейса: **русский**.

---

## 2. Пользователи и роли

| Роль | Возможности |
|------|-------------|
| **Обычный пользователь** | Просмотр и управление своими проектами/экспериментами |
| **Владелец проекта** | Управление участниками проекта (приглашение/удаление) |
| **Администратор** | Дополнительный раздел «Администрирование»: управление пользователями, сброс паролей, создание invite-токенов |

---

## 3. Структура приложения

### Страницы без авторизации

- **`/login`** — вход (username + password, ссылка на регистрацию)
- **`/register`** — регистрация (username + password)

### Основной лейаут (авторизованные)

```
┌─────────────────────────────────────────────────────┐
│  HEADER: [Название страницы]       [username] [Выйти]│
├───┬─────────────────────────────────────────────────┤
│   │                                                 │
│ S │           CONTENT AREA                          │
│ I │                                                 │
│ D │                                                 │
│ E │                                                 │
│ B │                                                 │
│ A │                                                 │
│ R │                                                 │
│   │                                    [FAB +]      │
└───┴─────────────────────────────────────────────────┘
```

- **Sidebar** — появляется при наведении мыши (hover reveal), ширина в скрытом
  состоянии минимальная, при наведении раскрывается поверх контента.
- **Header** — фиксированный, содержит название текущего раздела и кнопку пользователя.
  Клик по имени открывает модалку профиля (смена пароля).
- **FAB (Floating Action Button)** — кнопка «+» прибита к правому нижнему углу экрана,
  открывает модалку создания объекта текущего раздела.

### Разделы навигации

| Раздел | Путь | Описание |
|--------|------|----------|
| Проекты | `/projects` | Список проектов пользователя |
| Эксперименты | `/experiments` | Главный рабочий раздел |
| Датчики | `/sensors` | IoT-устройства |
| Телеметрия | `/telemetry` | Визуализация данных |
| Webhooks | `/webhooks` | Интеграции по событиям |
| Администрирование | `/admin/users` | Только для is_admin |

## 4. Детальное описание страниц

### 4.1 Login / Register

Центрированная карточка на весь экран.

**Login:**
- Заголовок «Вход в систему»
- Подзаголовок «Experiment Tracking Platform»
- Поля: Имя пользователя, Пароль
- Кнопка «Войти» (во время отправки: «Вход...», disabled)
- Ссылка «Нет аккаунта? Зарегистрироваться»

**Register:**
- Аналогичная форма с полями регистрации
- Ссылка на Login

---

### 4.2 Список проектов (`/projects`)

Сетка карточек проектов.

**Карточка проекта:**
- Заголовок (название проекта)
- Описание (если есть)
- Дата создания
- Кнопки действий (видны только владельцу):
  - ✏️ — открыть и редактировать (только для owner) или ℹ️ — только просмотр
  - 👥 — управление участниками (только для owner)

**Пустое состояние:** текст + кнопка «Создать первый проект».

**FAB:** открывает модалку создания проекта.

**Модалка проекта (create / view / edit):**
- Название (обязательно)
- Описание
- Участники (в режиме edit/view)

**Модалка управления участниками:**
- Список текущих участников с ролями (owner / editor / viewer)
- Добавление нового участника (username + выбор роли)
- Удаление участника (кроме owner)

---

### 4.3 Список экспериментов (`/experiments`)

Основная страница продукта. Сетка карточек с фильтрами.

**Фильтры (карточка сверху, горизонтальная сетка):**
- Текстовый поиск (по названию и описанию)
- Дропдаун «Проект» — автоматически выбирается первый проект при загрузке
- Дропдаун «Статус»: Все / Создан / Выполняется / Завершён / Ошибка / Архивирован
- Кнопки экспорта: CSV, JSON (disabled если нет projectId)

**Сетка экспериментов:**
- Карточки-ссылки на `/experiments/:id`
- Каждая карточка содержит:
  - Название + StatusBadge
  - Описание (если есть)
  - Тип эксперимента (если есть)
  - Теги
  - Дата создания

**Пагинация** (20 элементов на странице).

**FAB:** открывает модалку создания эксперимента.

**Модалка создания эксперимента:**
- Название (обязательно)
- Описание
- Тип
- Теги
- Метаданные (JSON)
- Выбор проекта

---

### 4.4 Детали эксперимента (`/experiments/:id`)

Разбита на два блока.

**Блок 1 — Информация об эксперименте:**
- Название + StatusBadge
- Кнопки: «Редактировать», «Удалить» (confirm-диалог)
- Описание
- InfoRow-список: ID, Project ID, Тип, Создан, Обновлён
- Теги
- Метаданные (pre-блок с JSON)

**Блок 2 — Запуски (Runs):**
- Заголовок «Запуски эксперимента» + кнопка «Новый запуск»
- Таблица запусков (компонент RunsList):
  - Чекбоксы для множественного выбора
  - Колонки: Название, Статус, Теги, Параметры (preview первых 3), Начало, Завершение, Длительность, Действия
  - Кнопки: «Bulk tagging (N)», «Экспорт CSV», «Экспорт JSON»

---

### 4.5 Детали запуска (`/runs/:id`)

Самая насыщенная страница.

**Блок — Информация о Run:**
- Название + StatusBadge
- Ссылка на родительский эксперимент
- InfoRow: ID, Experiment ID, Создан, Запущен, Завершён, Длительность
- Теги
- Параметры (key-value)
- Кнопки управления состоянием (зависят от текущего статуса):
  - «Запустить» (created → running)
  - «Завершить» (running → completed)
  - «Пометить как failed» (+ поле для причины)

**Блок — Метрики (RunMetrics):**
- Таблица или список метрик с числовыми значениями
- Шаги (step), временны́е метки

**Блок — Capture Sessions (сессии захвата данных):**
- Список сессий: номер, статус, длительность, заметки
- Кнопка «Начать отсчёт» (создаёт новую сессию)
- Для активной сессии: кнопки «Стоп» и «Стрим телеметрии»
- Для завершённых: кнопка «Удалить», «Экспорт телеметрии»
- Выбор датчика для сессии (дропдаун)

**Блок — Телеметрия:**
- Кнопка «Стрим» открывает TelemetryStreamModal (WebSocket, live-данные)
- Кнопка «Экспорт» открывает TelemetryExportModal

**Блок — Аудит-лог:**
- Хронологический список событий над этим Run

---

### 4.6 Датчики (`/sensors`)

Сетка карточек с фильтрами.

**Фильтры:**
- Дропдаун «Проект» (опционально, «Все проекты»)
- Дропдаун «Статус»: Все / Регистрация / Активен / Неактивен / Архивирован

**Карточка датчика (кликабельна → открывает SensorDetailModal):**
- Название + StatusBadge
- Тип
- Единицы измерения: input_unit → display_unit
- Последний heartbeat (относительное время: «2 мин назад», «Никогда»)
- Токен (маскированный: `****abc`)
- Дата создания

**FAB:** переход на `/sensors/new`.

**Модалка деталей датчика:**
- Полная информация о датчике
- Конфигурация

---

### 4.7 Создание датчика (`/sensors/new`)

Отдельная страница (не модалка).

Форма:
- Название
- Тип
- Единицы измерения (input / display)
- Описание / метаданные

---

### 4.8 Телеметрия (`/telemetry`)

Наиболее сложная страница с визуализацией. Plotly-графики.

**Два режима:**
- **Live** — живой стрим данных с датчика в реальном времени
- **History** — исторические данные по capture session

**Панель фильтров:**
- Проект → Эксперимент → Run → Capture Session
- Датчики (множественный выбор)
- Режим значений: physical / raw
- Включить поздние данные (include_late)
- Лимит точек (до 20 000)
- Порядок: asc / desc

**График:** линейный, каждый датчик своим цветом, интерактивный (Plotly).

**TelemetryPanel** — компонент с live-данными, показывает текущие значения в реальном времени.

**FAB:** открывает TelemetryExportModal.

---

### 4.9 Webhooks (`/webhooks`)

Страница разделена на два раздела.

**Раздел 1 — Подписки:**
- Список карточек подписок:
  - URL назначения
  - Статус (Активен/Неактивен) — бейдж
  - Типы событий — набор бейджей (run.started, run.finished, capture_session.created и т.д.)
  - Наличие secret (показывает `****`)
  - Кнопка «Удалить»
- Форма создания (раскрывается по кнопке «Создать»):
  - Target URL
  - Типы событий (через запятую, с подсказкой доступных)
  - Secret (опционально)

**Раздел 2 — История доставок:**
- Таблица: Статус, Тип события, URL, Попытки, Ошибка, Время, Действия
- Статусы: delivered (зелёный), failed (красный), pending (жёлтый), retrying
- Кнопка «Retry» для failed
- Фильтр по статусу
- Постраничная навигация

---

### 4.10 Администрирование (`/admin/users`)

Только для is_admin. Две вкладки.

**Вкладка «Пользователи»:**
- Поиск по username
- Фильтр: все / активные / неактивные
- Таблица пользователей:
  - Username, Email, Дата регистрации
  - Флаги: is_active (toggle), is_admin (toggle)
  - Действия: «Сбросить пароль» (показывает новый пароль в модалке), «Удалить»

**Вкладка «Invite-токены»:**
- Чекбокс «Только активные»
- Список токенов: код, email-подсказка, истекает, использован
- Кнопка копирования токена
- Форма создания: email-подсказка, срок действия (часы)

## 5. Компонентная система (UI Kit)

### StatusBadge

Цветные бейджи. Три набора статусов:

| Набор | Значения |
|-------|----------|
| **Эксперименты** | created (серый), running (синий), completed (зелёный), failed (красный), archived (серый) |
| **Runs** | created, running, completed, failed, archived — аналогично |
| **Датчики** | registering (жёлтый), active (зелёный), inactive (серый), archived (серый) |
| **Capture Sessions** | created, running, backfilling, stopped, archived |
| **Webhook Deliveries** | pending (жёлтый), delivered (зелёный), failed (красный), retrying (оранжевый) |

### Tags

Горизонтальный ряд мелких таблеток с текстом тегов.

### InfoRow

Строка `Метка: Значение`. Используется для мета-информации сущностей.
Метка — нейтральный цвет, значение — основной. UUID выводится моноширинным шрифтом.

### EmptyState

Центрированный блок: иконка/иллюстрация, текст, опциональная кнопка-призыв к действию.

### Loading

Индикатор загрузки с опциональным текстом («Загрузка проектов...»).

### FloatingActionButton (FAB)

Круглая кнопка «+», фиксированная в правом нижнем углу экрана.

### Pagination

Строка с кнопками страниц под списком.

### MaterialSelect

Стилизованный `<select>` с float-label (Material Design стиль).

### Modal

Базовый компонент модального окна. Используется везде для создания/редактирования.

### FormGroup

Обёртка для `label + input`, стандартный отступ.

### FormActions

Строка кнопок «Отмена / Сохранить» внизу формы.

### Toast-уведомления

Всплывающие уведомления в углу экрана:
- Успех (зелёный)
- Ошибка (красный) — с опциональными деталями ошибки для отладки

### PageHeader

Заголовок страницы с опциональными действиями (кнопки справа).

## 6. Паттерны взаимодействия

### Создание объектов
- Везде через FAB в нижнем правом углу → открывает модалку
- Исключение: создание датчика — отдельная страница `/sensors/new`

### Редактирование
- Кнопка «Редактировать» на странице деталей → модалка поверх страницы

### Удаление
- Кнопка «Удалить» + `confirm()`-диалог браузера
- Toast «Успешно удалено» + редирект (если удалена текущая страница)

### Управление состоянием (State Machine)
Runs и Experiments имеют конечный автомат состояний.
UI показывает только допустимые переходы в виде кнопок.

### Экспорт данных
- Кнопки CSV / JSON прямо на странице списка (или в шапке RunsList)
- Скачивание через `<a download>`, без перехода на новую страницу

### Реальное время
- TelemetryStreamModal: WebSocket-подключение, live-обновление значений
- TelemetryPanel: отображает текущие значения датчиков

## 7. Контентные особенности

| Тип данных | Пример | Где встречается |
|-----------|--------|-----------------|
| UUID | `3f2504e0-4f89-11d3-9a0c-0305e82c3301` | ID сущностей, моноширинный шрифт |
| JSON-объект | `{"lr": 0.001, "batch": 32}` | Параметры, метаданные — pre-блок |
| Временны́е метки | `15 фев 2026 14:32` | Везде, русский формат |
| Относительное время | `2 мин назад`, `Никогда` | Heartbeat датчиков |
| Длительность | `2ч 15м 3с` | Run duration, Session duration |
| Числовые метрики | `accuracy: 0.9234` | RunMetrics |
| Телеметрия | Float-числа с временны́ми метками | TelemetryViewer (Plotly) |
| URL | `https://example.com/webhook` | Webhooks |
| Токен (маскированный) | `****abc1` | Sensor token |

## 8. Технические ограничения для дизайнера

- **Компонентная библиотека:** MUI (Material UI) используется для отдельных элементов
  (Select, возможно кнопки). Остальное — кастомный SCSS.
- **Стили:** SCSS-модули, можно менять полностью.
- **Темизация:** сейчас только светлая тема.
- **Графики:** Plotly.js (интерактивные линейные графики, внешний вид поддаётся кастомизации).
- **Шрифт:** сейчас не задан явно (системный стек). UUID/код — моноширинный.
- **Адаптивность:** неизвестно, требуется ли мобильная версия (аудитория — desktop).
- **Иконки:** сейчас emoji (✏️, ℹ️, 👥) — требуют замены на иконочный шрифт или SVG.
- **Анимации:** минимальные (page-transition класс на смену страниц).

## 9. Что точно нужно переработать (известные боли)

1. **Sidebar hover** — неудобно, часто случайно открывается. Рассмотреть постоянный collapsed sidebar с иконками.
2. **Emoji-кнопки** (✏️, 👥) — непрофессионально, заменить на иконки.
3. **confirm() для удаления** — системный диалог браузера, нужна кастомная модалка подтверждения.
4. **Таблица Runs** — много колонок, часть данных обрезается. Нужна продуманная компоновка.
5. **Webhook-доставки** — таблица с длинными URL без переноса, неудобна.
6. **Нет единой цветовой системы** — статусы, бейджи, кнопки определены разрозненно.
7. **Loading/Empty states** — нет иллюстраций, только текст.
8. **Форма создания Webhook** — раскрывающаяся inline-форма выглядит как временное решение.
