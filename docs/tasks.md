## RFC: вычисляемые/производные величины (computed channels)

**Приоритет:** MEDIUM
**Статус:** [ ] Подготовка RFC

### Мотивация

Во многих экспериментах физические величины не снимаются напрямую, а вычисляются из показаний
нескольких датчиков. Примеры:

- **Число Маха** в аэродинамической трубе: функция от давления и температуры.
- **Массовый расход воздуха**: интеграл по времени от произведения плотности и скорости потока.
- **Суммарная энергия, импульс**: интегральные величины, которые нужно периодически «обнулять»
  (между запусками или capture sessions).
- **Производные по времени** (dP/dt): численное дифференцирование показаний.

Сейчас в платформе есть только per-sensor преобразование `raw → physical` через
`conversion_profiles` (`linear`, `polynomial`, `lookup_table`) — см. `backend_common/conversion.py`.
Оно не покрывает мульти-сенсорные формулы и работу с рядами значений.

### Идеи реализации

Способы задания формулы — от простого к сложному:

1. **Арифметические формулы** по последним значениям нескольких датчиков
   (`mach = f(pressure, temperature)`). Реализуется расширением существующих
   `conversion_profiles` новым `kind = "computed"` с `payload.inputs = [sensor_id, signal]` и
   `payload.expression`. Требует безопасного парсера выражений (AST whitelist, см. фаза 4.3.3
   в roadmap — «Custom formula»).

2. **Агрегации по окну** (скользящее среднее, интегралы, производные): нужен stateful computе
   с буфером последних N точек/секунд. Сбрасывать состояние по событиям
   (`capture_session.started/stopped`, `run.started/completed`).

3. **Пользовательский DSL или скриптовый движок** — максимальная гибкость, доступ к рядам
   значений. Минусы: порог входа для оператора, safety sandbox. Возможна интеграция с уже
   существующим **Script Runner** (RabbitMQ, `backend_common/script_runner/`).

### Архитектурные вопросы

- **Где считать:** внутри `telemetry-ingest-service` (низкая latency, но нагружает самый
  нагруженный сервис) или отдельный **computed-channels-service** (отвязка, но +hop).
- **Стек:** Python достаточно или для горячего пути нужен C++/Rust?
  (Сейчас telemetry-ingest на Python/aiohttp — если станет bottleneck, это главный кандидат
  на переписывание.)
- **Хранение:** результаты писать как обычный `telemetry_records` с `sensor_id = <virtual>`
  или завести отдельную сущность «computed channel»?
- **Обратная совместимость:** как отображать computed channels в UI (`TelemetryViewer`,
  экспорт) — как обычные сенсоры или отдельным слоем?

### Связь с существующим

- **Conversion profiles** (реализовано): per-sensor `raw → physical`.
- **Script Runner** (реализовано): может быть использован для вычислений по расписанию
  или по событию.
- **Зависимость:** решение по этой задаче повлияет на архитектуру алертов (задача ниже) —
  пороговые значения часто ставят именно на производные величины.

## RFC: алерты по значениям датчиков

**Приоритет:** LOW
**Статус:** [ ] Подготовка RFC

### Мотивация

При проведении эксперимента оператор должен вовремя замечать выход значений за критические
пороги (давление, температура, вибрации) — иначе можно пропустить аварийную ситуацию или
повредить оборудование. Сейчас в UI только графики — на них легко пропустить редкий выброс,
особенно при многочасовых прогонах.

### Use cases

- «Давление в трубе не должно превышать X бар — иначе остановить прогон».
- «Температура обмотки выше Y °C — включить индикатор и звуковой сигнал».
- «Вибрация в допустимых пределах — зелёный, предупреждение — жёлтый, критично — красный».

### Идеи реализации

**Backend:**
- Таблица `sensor_thresholds` (sensor_id, signal, rules: `[{op, value, severity, color}]`).
- Движок событий: при ingest сравнивает с порогами, публикует события алертов.
- Хранение истории срабатываний (для аудита и пост-анализа).
- Опционально: пороги по computed channels (см. задачу выше).

**Frontend:**
- Новые виджеты для `/telemetry`:
  - **Лампочка** (цвет по порогу, зелёный / жёлтый / красный, настраиваемые цвета).
  - **Стрелочный прибор** (gauge) с диапазонами.
  - **Цифровой индикатор** с подсветкой при выходе за порог.
  - **Текстовый блок** с шаблоном (`"{signal}: {value} {unit}"`).
- **Звуковые сигналы** при переходе в критический диапазон (с возможностью заглушить).
- **Dashboard-конструктор** для `/telemetry`:
  - Сохранять набор панелей в БД (dashboard → panels).
  - Несколько dashboards для пользователя/проекта.
  - Возможность открыть разные dashboards в отдельных окнах браузера (для нескольких мониторов).

### Открытые вопросы

- Уведомления за пределами UI: webhook/email/telegram — нужно ли в MVP?
- Quiet hours / подтверждение оператором (acknowledge) — часть MVP или v2?
- Иерархия: пороги на датчик / проект / глобально?

### Связь с существующим

- **Webhook dispatcher** (реализовано): можно переиспользовать для доставки событий алертов.
- **Audit log** (реализовано): история срабатываний пишется сюда.
- **Computed channels** (задача выше): пороги часто ставят на производные величины —
  порядок реализации: computed channels → алерты.

## RFC: система управления экспериментом (closed-loop control)

**Приоритет:** LOW
**Статус:** [ ] Подготовка RFC

### Мотивация

Помимо наблюдения (алерты), в некоторых экспериментах нужно автоматически воздействовать
на стенд: открыть/закрыть клапан, изменить обороты вентилятора, включить нагреватель.
Это шаг от observability к control.

### Варианты подхода

- **Open-loop:** оператор вручную шлёт команды из UI. Проще реализовать — в сущности это
  расширение API и UI кнопками/слайдерами.
- **Closed-loop:** автоматические действия по правилам («если давление > X — открыть клапан»).
  Сложнее: нужно гарантировать latency, устойчивость к потере связи, отсутствие автоколебаний.

### Ключевые риски

- **End-to-end latency:** датчик → ingest → БД → rule engine → команда → устройство.
  Сейчас даже без control это десятки мс из-за HTTP-стэка. Для быстрых процессов
  (≥ 10 Гц управления) это уже критично. Для медленных (нагрев, давление) — приемлемо.
- **Фазовая задержка** в обратной связи → риск колебаний / неустойчивости контура. Нужно
  заложить pole placement / PID tuning или ограничиться релейным управлением с гистерезисом.
- **Безопасность:** failsafe, watchdog, подтверждение критических команд, изоляция
  пользовательских правил (sandbox).

### Применимость

- **Хорошо:** медленные процессы (температура, уровень жидкости), где допустимо
  время реакции в секундах.
- **Плохо / не в scope:** быстрая динамика (вибрации, аэродинамика) — там нужен
  локальный контроллер на самом стенде (как в RC Vehicle с циклом 500 Гц).

### Связь с существующим

- **Алерты** (задача выше): closed-loop — это алерт + action. Делать **строго после**
  алертов, переиспользуя движок правил.
- **Script Runner** (реализовано): может быть бэкендом для выполнения action-скриптов
  с ограничениями по времени.

## Нагрузочное тестирование: запуск и автоматизация

**Приоритет:** HIGH
**Статус:** [ ] Подготовка

### Мотивация

Платформа ещё ни разу не тестировалась под нагрузкой. Не известны:
- Максимальный throughput по ingest (REST / WebSocket).
- Поведение SSE при большом числе одновременных подписчиков.
- Деградация API при параллельных экспортах / запросах агрегатов.
- DB-лимиты (connection pool, checkpoints на TimescaleDB).

Цель roadmap (P95 API < 400 мс при 200 RPS, 200 одновременных датчиков, 5k точек/сек)
не подтверждена измерениями. Без нагрузочного тестирования регрессии по perf будут
ловиться на проде.

### Идеи реализации

**Частично уже спроектировано** в [experiment-tracking-status-and-roadmap.md](experiment-tracking-status-and-roadmap.md),
раздел «Песочница для нагрузочного тестирования» (инструмент k6, сценарии ingest / API /
SSE / export, Docker Compose profile `load-test`, k6 dashboard в Grafana).

Эта задача — **запустить этот план** и дополнить его процессной частью:

1. **Реализовать k6-сценарии** (ingest_load, api_load, sse_load, export_load).
2. **Seed-скрипт** для предзаполнения БД (N датчиков, M экспериментов, K точек телеметрии).
3. **Makefile-цель** `make load-test` — одна команда, поднимает стек + запускает прогон.
4. **Регулярный запуск** — nightly / по тегу релиза (GitHub Actions job с публикацией
   отчёта в артефакты).
5. **Baseline + regression gates** — зафиксировать текущие P95 и падать CI при деградации
   > N%.

### Открытые вопросы

- Тестировать в Docker Compose (быстро, но не репрезентативно) или поднимать отдельный
  стенд в Yandex Cloud (медленнее, но реалистичнее)?
- Нужны ли профили нагрузок под разные сценарии (RC-машинка 500 Гц vs лабораторный
  стенд 10 Гц)?

### Связь с существующим

- **Grafana + Prometheus** (реализовано): метрики уже собираются, для теста нужен
  только дашборд нагрузки.
- **OpenTelemetry** (реализовано): трейсы помогут найти узкие места.
