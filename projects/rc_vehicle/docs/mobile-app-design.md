## 1. Выбор технологии

### 1.1 Сравнение вариантов

| Критерий | Flutter | React Native | Kotlin/Swift (нативное) |
|---|---|---|---|
| Латентность ввода (touch) | ~1-2 мс (Skia рендер, минимум прослоек) | ~5-10 мс (JS bridge, хотя Fabric/JSI сокращают) | <1 мс (нативные view) |
| WebSocket | `web_socket_channel` (стабильная) | `react-native-websocket` или Fetch API | Нативные API, полный контроль |
| UDP сокеты | `dart:io` UdpSocket -- работает | `react-native-udp` (community, нестабильна) | Нативные API, полный контроль |
| Гироскоп/акселерометр | `sensors_plus` | `react-native-sensors` | Нативные API |
| Haptic feedback | `HapticFeedback` API (ограничено) | `react-native-haptic-feedback` | UIFeedbackGenerator / VibrationEffect (полный контроль) |
| Background WebSocket | Dart isolate, работает | Ограничено (JS thread suspends) | Foreground service / BGTaskScheduler |
| Кроссплатформенность | iOS + Android из одной кодовой базы | iOS + Android из одной кодовой базы | Две отдельные кодовые базы |
| Сложность для одного студента | Средняя (Dart -- новый язык, но экосистема зрелая) | Низкая если есть опыт React (есть -- фронтенд проекта на React) | Высокая (два языка, две платформы) |
| Оффлайн/бинарный UDP | Dart `ByteData`, `struct`-like парсинг | Сложно, JS не дружит с бинарными данными | Тривиально |

### 1.2 Рекомендация: Flutter

**Обоснование:**

1. **UDP поддержка из коробки.** `dart:io` дает `RawDatagramSocket` без сторонних библиотек. Для React Native нужен нестабильный community-модуль `react-native-udp`, который периодически ломается на новых версиях RN. Это критично -- UDP-телеметрия 100 Hz является ключевой фичей.

2. **Латентность ввода достаточна.** Для RC-машинки управляющие команды идут с частотой 50 Hz (20 мс между пакетами). Разница между 1 мс (нативное) и 2 мс (Flutter) нерелевантна при 20 мс квантовании. Flutter рендерит canvas-виджеты (джойстик) без bridge overhead.

3. **Один язык -- один разработчик.** Dart проще для студента, чем поддерживать две нативные кодовые базы. React Native был бы логичен при наличии опыта React, но проблема с UDP перевешивает.

4. **Бинарный парсинг.** `ByteData` и `Endian.little` в Dart позволяют парсить 79-байтные UDP-пакеты телеметрии без сторонних зависимостей. В JS/React Native это потребует `ArrayBuffer` + community-пакеты для UDP, что ненадежно.

5. **Зрелые библиотеки сенсоров.** `sensors_plus` (гироскоп для gyro-steering) и `HapticFeedback` покрывают нужды приложения.

**Компромисс:** Dart -- новый язык (не используется в остальном проекте). Но он прост, а Flutter-экосистема для мобильных приложений зрелее, чем React Native для низкоуровневых задач (UDP, бинарные протоколы).

## 2. Архитектура приложения

### 2.1 Экраны и навигация

```
App
├── ConnectionScreen          # Поиск и подключение к ESP32
├── MainScreen (tabs)
│   ├── DriveTab              # Управление (landscape)
│   ├── TelemetryTab          # Графики в реальном времени
│   ├── SetupTab              # Стабилизация, режимы, калибровка
│   └── DiagnosticsTab        # Self-test, Wi-Fi STA, логи
└── SettingsScreen            # Настройки приложения (не машины)
```

**Навигация:**
- `ConnectionScreen` -- стартовый. После подключения переход на `MainScreen`.
- `MainScreen` -- bottom tab bar. При переходе на `DriveTab` -- автоматический переход в landscape и скрытие tab bar.
- `DriveTab` -- полноэкранный, landscape-only. Выход -- свайп или кнопка "назад".

### 2.2 State management

**Riverpod** (Flutter). Провайдеры:

| Провайдер | Тип | Назначение |
|---|---|---|
| `connectionProvider` | `StateNotifier` | Состояние подключения (disconnected/connecting/connected), IP адрес ESP32 |
| `wsProvider` | `StateNotifier` | WebSocket: отправка команд, получение JSON-телеметрии (20 Hz) |
| `udpTelemetryProvider` | `StreamProvider` | UDP-стрим бинарной телеметрии (100 Hz) |
| `driveInputProvider` | `StateNotifier` | Текущие throttle/steering от джойстика или гироскопа |
| `stabConfigProvider` | `StateNotifier` | Конфигурация стабилизации (кэш + синхронизация с ESP32) |
| `vehicleStateProvider` | `Provider` | Агрегированное состояние: режим, failsafe, kids mode, battery |

```
                  │
           ┌──────▼──────┐
           │   ESP32-S3  │
```

**WebSocket** (`ws://<ip>/ws`):
- Управляющие команды: `throttle,steering` (50 Hz).
- JSON-команды: калибровка, конфиг стабилизации, self-test, kids mode.
- JSON-телеметрия (20 Hz) -- для отображения на экранах Setup/Diagnostics.

**UDP** (порт 5555):
- Бинарная телеметрия 100 Hz -- для записи и графиков реального времени.
- Приложение отправляет `START <port>` на порт 5556 при переходе на TelemetryTab.
- `STOP` при уходе с экрана или сворачивании приложения.

**REST** (`http://<ip>/api/wifi/*`):
- WiFi STA management (scan, connect, disconnect) -- редкие операции.

### 2.4 Reconnect стратегия

```
Потеря WebSocket
    │
    ├── Немедленно: throttle=0, steering=0 (на стороне приложения)
    │   ESP32 сам перейдет в failsafe через 250 мс
    │
    ├── Попытка 1: через 500 мс
    ├── Попытка 2: через 1 с
    ├── Попытка 3: через 2 с
    ├── ...exponential backoff до 5 с max
    │
    ├── После 10 неудачных попыток:
    │   показать экран переподключения,
    │   проверить WiFi-соединение телефона
    │
    └── При восстановлении:
        запросить get_stab_config для синхронизации состояния
```

Для UDP: потеря пакетов допустима (телеметрия). Детекция через sequence number gap. При gap > 100 (1 секунда) -- warning в UI.

### 3.2 Способы ввода

**Рекомендация: раздельные зоны (throttle слева + steering джойстик справа).**

Сравнение вариантов:

| Вариант | Плюсы | Минусы |
|---|---|---|
| Двухосевой джойстик | Привычен геймерам | Сложно управлять газом и рулём одновременно одним пальцем. Нет разделения газ/тормоз |
| Раздельные зоны | Как на RC-пульте (левый стик = газ, правый = руль). Естественно для RC. Раздельный газ/тормоз | Занимает больше места |
| Слайдеры | Просто в реализации | Неудобно на телефоне (текущая проблема веб-UI) |
| Тачпад (тап = направление) | Минималистично | Неточно, нет пропорционального управления |

**Левая зона (throttle/brake):**
- Вертикальный тачпад, не слайдер.
- Касание выше центра = gas (0..+1), ниже = brake (0..-1). Центр = нейтраль.
- Отпускание пальца = мгновенный возврат в нейтраль (0).
- Визуальная полоса показывает текущее значение.

**Правая зона (steering):**
- Однокоординатный горизонтальный джойстик (круг, который двигается влево-вправо).
- Касание + drag = поворот (-1..+1).
- Отпускание = возврат в центр (0).
- Можно расширить до двухосевого джойстика (вертикаль = trim или дополнительная ось), но для MVP не нужно.

**Почему не двухосевой джойстик для обоих:**
RC-пульт имеет два отдельных стика именно потому, что одновременное пропорциональное управление газом и рулём одной рукой неточно. Раздельные зоны = две руки = два независимых пальца.

### 3.3 Gyro-steering (опционально)

- Наклон телефона влево/вправо = steering (-1..+1).
- Включается кнопкой `[Gyro]` на экране управления.
- При активации: правая зона скрывается, вместо нее -- индикатор текущего угла наклона.
- Левая зона (throttle/brake) остается тачпадом.
- Калибровка нуля: текущее положение телефона при нажатии `[Gyro]` = нейтраль.
- Зона нечувствительности (deadzone): +/- 3 градуса от нуля.
- Максимальный угол: +/- 30 градусов = полный поворот.
- Фильтрация: LPF на данных гироскопа телефона (убрать дрожание рук). Cutoff ~5 Hz.

### 3.4 Haptic feedback

| Событие | Паттерн вибрации | Интенсивность |
|---|---|---|
| Failsafe активирован | 3 коротких импульса (100мс on, 100мс off) | Сильная |
| Потеря WebSocket | Длинная вибрация (500 мс) | Сильная |
| Занос (oversteer_active = 1.0) | Непрерывная лёгкая вибрация пока занос активен | Слабая |
| Kids Mode: достигнут предел газа | 1 короткий импульс (50 мс) | Средняя |
| Kids Mode: достигнут предел руля | 1 короткий импульс (50 мс) | Средняя |
| Смена режима (Normal/Sport/Drift/Kids) | 2 коротких импульса | Лёгкая |

Реализация: `HapticFeedback.heavyImpact()` / `lightImpact()` / `vibrate()` из Flutter. На Android дополнительно `VibrationEffect.createWaveform()` через platform channel для паттернов.

### 3.5 Lock-on-screen

- При входе на DriveTab:
  - Ориентация = landscape (принудительно).
  - Wake Lock активен (экран не гаснет).
  - Системные жесты отключены (edge-to-edge, иначе swipe-back от iOS/Android мешает).
  - Status bar скрыт (immersive mode).
- При выходе (кнопка "назад" на экране, или hardware back) -- всё возвращается.

### 3.6 Частота отправки команд

- 50 Hz (каждые 20 мс) -- совпадает с текущим WebSocket протоколом.
- Если значение throttle/steering не изменилось -- всё равно отправлять (keepalive для failsafe).
- Формат: `"<throttle>,<steering>\n"` -- текущий протокол ESP32.

### 4.1 ConnectionScreen

**Назначение:** Поиск ESP32 и подключение.

**Отличие от веба:** В браузере пользователь вручную набирает IP. В приложении -- автоматическое обнаружение.

**Логика обнаружения:**

1. **Проверка WiFi.** Если телефон подключен к сети с SSID, содержащим "RC" или "ESP" -- попробовать gateway IP (обычно 192.168.4.1 в AP режиме).
2. **Фиксированный IP.** Кнопка "Подключиться к 192.168.4.1" (AP режим ESP32 всегда использует этот адрес).
3. **Ручной ввод IP.** Для STA режима, когда ESP32 получает IP от роутера.
4. **История подключений.** Список последних 5 успешных IP, сохраненный в SharedPreferences.

**Проверка связи:** WebSocket ping на `ws://<ip>/ws`. Если ответ -- показать "Подключено", перейти на MainScreen.

### 4.2 TelemetryTab

**Назначение:** Графики в реальном времени + запись + экспорт.

**Источник данных:** UDP 100 Hz (бинарный стрим). При входе на вкладку приложение отправляет `START 5555 100` через UDP порт 5556. При уходе -- `STOP`.

**Графики (scrolling, последние 10 секунд):**

| График | Данные | Цвет |
|---|---|---|
| Ориентация | pitch, roll, yaw (deg) | R, G, B |
| Угловая скорость | gx, gy, gz (dps) | R, G, B |
| Ускорение | ax, ay, az (g) | R, G, B |
| Скорость и занос | speed_ms, slip_deg | Синий, Красный |
| Управление | throttle, steering | Зелёный, Оранжевый |

**Отличие от веба:**
- Веб показывает 20 Hz JSON-телеметрию. Приложение -- 100 Hz UDP бинарные данные. В 5 раз больше точек, более гладкие графики.
- Веб может экспортировать 500 кадров из ring buffer. Приложение пишет непрерывно на телефон -- ограничено только памятью.

**Запись:**
- Кнопка `[REC]` -- начать запись в локальный файл.
- Кнопка `[STOP REC]` -- остановить.
- Индикатор: количество записанных кадров, размер файла, длительность.
- Формат: CSV (совместимый с Python-скриптом `udp_telem_receiver.py`).
- Хранение: `getApplicationDocumentsDirectory()` / `telemetry/` / `YYYY-MM-DD_HH-MM-SS.csv`.

**Экспорт:** Share sheet (iOS/Android) для отправки CSV по email, Telegram, AirDrop.

**Статистика (снизу экрана):**
- Packet rate: X Hz (реальная частота получения).
- Lost packets: N (по sequence number).
- Loss %: X.X%.

**Рендеринг графиков:**
Библиотека `fl_chart` (Flutter). Оптимизация: хранить в памяти только последние 1000 точек (10 сек при 100 Hz). Для записи в файл -- все данные.

## 5. Телеметрия и запись

### 5.1 Архитектура приёма UDP

```
┌──────────────┐     UDP 100Hz      ┌──────────────────┐
│   ESP32-S3   │ ──── 79 bytes ───→ │  UdpReceiver     │
│  port 5555   │                    │  (Dart isolate)  │
└──────────────┘                    └────────┬─────────┘
                                             │ parsed TelemetryFrame
                                    ┌────────▼─────────┐
                                    │   StreamController │
                                    └──┬─────────┬──────┘
                                       │         │
                              ┌────────▼──┐  ┌───▼────────┐
                              │  UI Charts │  │  CSV Writer │
                              │ (last 10s) │  │ (if recording)│
                              └───────────┘  └────────────┘
```

**Dart Isolate для UDP:**
Приём UDP-пакетов происходит в отдельном Dart isolate, чтобы не блокировать UI thread. Isolate:
1. Открывает `RawDatagramSocket` на порте 5555.
2. Парсит каждый пакет: проверяет magic bytes (0x52, 0x54), версию, извлекает `TelemetryLogFrame`.
3. Отслеживает sequence number для детекции потерь.
4. Отправляет распарсенный `TelemetryFrame` в основной isolate через `SendPort`.

### 5.2 Парсинг пакета (Dart)

```dart
class TelemetryFrame {
  final int seqNum;
  final int tsMs;
  final double ax, ay, az;       // accel [g]
  final double gx, gy, gz;       // gyro [dps]
  final double vx, vy;           // EKF velocity [m/s]
  final double slipDeg;          // slip angle [deg]
  final double speedMs;          // speed [m/s]
  final double throttle;         // [-1..1]
  final double steering;         // [-1..1]
  final double pitchDeg, rollDeg, yawDeg;
  final double yawRateDps;
  final bool oversteerActive;

  factory TelemetryFrame.fromBytes(Uint8List data) {
    // data.length == 87
    // [0..1] = magic 0x52, 0x54
    // [2]    = version 0x01
    // [3..6] = seq (uint32 LE)
    // [7..86] = TelemetryLogFrame (80 bytes, all LE)
    final bd = ByteData.sublistView(data);
    return TelemetryFrame(
      seqNum: bd.getUint32(3, Endian.little),
      tsMs: bd.getUint32(7, Endian.little),
      ax: bd.getFloat32(11, Endian.little),
      // ... и так далее для всех полей
    );
  }
}
```

### 5.3 Запись CSV

Заголовок CSV совместим с `udp_telem_receiver.py`:

```
ts_ms,ax,ay,az,gx,gy,gz,vx,vy,slip_deg,speed_ms,throttle,steering,pitch_deg,roll_deg,yaw_deg,yaw_rate_dps,oversteer_active
```

Запись: буферизация по 100 строк, flush на диск. При 100 Hz -- flush раз в секунду.

Размер: ~120 байт/строка x 100 Hz = 12 KB/с = ~43 MB/час. Допустимо для телефона.

### 5.4 Графики реального времени

Библиотека: `fl_chart` (LineChart).

Оптимизация:
- Кольцевой буфер на 1000 точек (10 сек при 100 Hz) для каждого графика.
- Перерисовка по `AnimationController` с частотой 30 FPS (не 100 -- UI не успеет).
- Downsampling для отображения: каждый 3-й или 4-й пакет (25-33 Hz visual update), но все 100 Hz пишутся в файл.

## 6. Связь и надёжность

### 6.1 Обнаружение ESP32

**Порядок попыток:**

1. **Fixed AP IP:** 192.168.4.1 (ESP32 в режиме AP всегда использует этот адрес). WebSocket ping, таймаут 2 с.
2. **Последний успешный IP** из SharedPreferences. WebSocket ping, таймаут 2 с.
3. **Ручной ввод** -- пользователь вводит IP.

mDNS не используется: ESP32-S3 с текущей прошивкой не публикует mDNS-запись, а добавление mDNS на ESP32 потребует изменений прошивки. Для одной машинки фиксированный IP (AP) + ручной ввод (STA) достаточно.

### 6.2 Индикация качества связи

**Status bar на DriveTab:**

| Индикатор | Источник | Отображение |
|---|---|---|
| WebSocket latency | Замер RTT: отправка ping-команды, ожидание pong | Зелёный <50мс, Жёлтый <150мс, Красный >150мс |
| UDP packet loss | Sequence number gaps за последние 5 секунд | 0% = зелёный, <1% = жёлтый, >1% = красный |
| Состояние подключения | WebSocket readyState | Зелёная/красная точка |

### 6.3 Reconnect при сворачивании

**Проблема:** При сворачивании приложения iOS/Android могут убить сетевые соединения.

**Решение:**
- При уходе в background (`WidgetsBindingObserver.didChangeAppLifecycleState`):
  - Отправить throttle=0, steering=0.
  - Отправить UDP `STOP` (если стриминг активен).
  - Не закрывать WebSocket явно (пусть ОС решает).
- При возврате в foreground:
  - Проверить WebSocket. Если закрыт -- reconnect.
  - Если был на TelemetryTab -- возобновить UDP стрим.
  - Запросить `get_stab_config` для синхронизации состояния.

### 6.4 Failsafe (безопасность)

Приложение не заменяет failsafe на ESP32, а дополняет его:
- ESP32 переходит в failsafe через 250 мс без WebSocket-команд (независимо от приложения).
- Приложение показывает баннер "FAILSAFE" при получении соответствующего статуса в телеметрии.
- При потере связи приложение показывает красный экран "Связь потеряна" и попытки reconnect.

## 7. MVP и этапы

### Этап 1: MVP -- Управление + базовая телеметрия (3-4 недели)

**Цель:** Замена веб-интерфейса для управления.

**Scope:**
- ConnectionScreen (фиксированный IP + ручной ввод).
- DriveTab: throttle/brake зона + steering джойстик (без gyro-steering).
- WebSocket: отправка команд 50 Hz, приём JSON-телеметрии 20 Hz.
- Status bar: скорость, режим, состояние подключения.
- Landscape + wake lock + immersive mode.
- Reconnect при разрыве WebSocket.

**Результат:** Можно управлять машинкой с телефона удобнее, чем из браузера.

### Этап 2: Телеметрия и запись (2-3 недели)

**Цель:** Полноценная система телеметрии с записью.

**Scope:**
- TelemetryTab с графиками реального времени.
- UDP-приём 100 Hz в отдельном isolate.
- Запись CSV на телефон.
- Экспорт CSV через share sheet.
- Статистика: packet rate, loss %, sequence gaps.

**Результат:** Телефон как портативный логгер телеметрии (замена Python-скрипта на ноутбуке).

### Этап 3: Настройки и диагностика (2-3 недели)

**Цель:** Полный аналог веб-интерфейса.

**Scope:**
- SetupTab: режимы, PID, калибровка, Kids Mode.
- DiagnosticsTab: self-test, WiFi STA.
- SettingsScreen (инверсия осей, deadzone, тема).
- Haptic feedback (failsafe, занос, пределы kids mode).

**Результат:** Веб-интерфейс больше не нужен для повседневного использования.

### Этап 4: Продвинутые фичи (2-3 недели)

**Цель:** Фичи, которых нет в вебе.

**Scope:**
- Gyro-steering (наклон телефона = руль).
- История подключений с автоматическим обнаружением.
- Overlay-режим телеметрии на DriveTab (полупрозрачные мини-графики поверх джойстика).
- Экспорт настроек стабилизации в JSON-файл (для переноса между машинками).
- Виджет быстрого подключения (Android: widget на рабочий стол).

**Результат:** Приложение превосходит веб-интерфейс по функциональности.
