# Идеи для улучшения прошивки

## 1. Авто-калибровка движения вперёд без фиксированного управляющего воздействия

**Проблема:** при разном заряде батареи одно и то же PWM даёт разную скорость,
поэтому авто-калибровка (`auto_forward`) с фиксированным `throttle` ненадёжна.

**Варианты решения:**

- **(a) Ручной разгон (рекомендуется для ближайшей итерации):** оператор газует сам,
  прошивка только собирает данные IMU. Нет зависимости от точности EKF.
- **(b) PID по forward_accel:** задавать не скорость, а целевое ускорение.
  Акселерометр не дрейфует (в отличие от интеграла скорости), но нужна фильтрация вибраций.
- **(c) Speed feedback:** замкнутый контур по `ekf.speed_ms` — возможен только после
  добавления энкодера/optical flow. Без них EKF `speed_ms` считается чистым
  интегрированием акселерометра и накапливает дрейф.

**Зависимости:** вариант (a) — нет, (b) — нет, (c) — энкодер.

## 2. Тестовые автономные режимы движения

Автономные паттерны для сбора данных и настройки:

- **Круг (постоянный):** максимальный поворот + медленный газ — стабильный занос для настройки EKF/slip
- **Круг с нарастанием газа:** постепенное ускорение до появления сноса передней/задней оси — определение предела сцепления
- **Максимальное ускорение:** полный газ в течение N секунд — для калибровки `forward_accel` и EKF

### Failsafe — критичная проблема

Failsafe срабатывает через **250 мс** если нет ни RC, ни WiFi-команд.
Автономный режим будет остановлен через четверть секунды при потере WiFi.

Необходимо:
- **WiFi heartbeat из Web UI** — пока UI открыт, отправлять keepalive каждые ~100 мс
- **Кнопка аварийной остановки** в UI для немедленной отмены автономного режима
- **Таймер максимальной длительности** автономного теста (например, 30 секунд) как дополнительная защита
- **Не ослаблять failsafe** — 250 мс таймаут остаётся как safety net, heartbeat не даёт ему сработать

### Телеметрия для тестовых прогонов

Создать отдельный буфер в PSRAM для записи длинных сессий.

Текущее состояние:
- Ring-buffer: 5000 фреймов × 80 байт = **400 КБ** (при 20 Гц = ~4 мин 10 сек записи)
- PSRAM: 16 МБ доступно

Можно увеличить до 50 000 фреймов (3.6 МБ, ~41 мин записи) — малая доля PSRAM.

**Зависимости:** heartbeat + failsafe доработка.

## 3. Детский режим: ограничение по скорости/ускорению вместо PWM

**Проблема:** при разном заряде батареи одно и то же ограничение throttle PWM даёт
разную реальную скорость — режим ненадёжен.

**Текущая реализация:** multiplier-clamp (`throttle = min(throttle, limit)`, default 0.3),
slew rate 0.3/сек, anti-spin по slip angle (EKF).

### Ограничение EKF

EKF `speed_ms` без энкодеров дрейфует, а ниже 0.3 м/с возвращает 0 —
для детского режима на малых скоростях это критично.

### Реалистичный промежуточный вариант

Ограничение по **forward_accel** (IMU, не дрейфует): если ускорение превышает порог,
throttle PWM снижается. Не гарантирует точную скорость, но ограничивает динамику
разгона независимо от заряда батареи.

Полноценное ограничение по скорости — только после добавления энкодера.

**Зависимости:** ограничение по accel — нет (IMU уже есть); ограничение по speed — энкодер.

## 4. Режимы торможения

- **Плавное торможение:** при отпускании курка до нейтрали — постепенное снижение скорости (мотор-тормоз или обратный PWM)
- **Свободный накат:** при отпускании курка — не тормозить, машина катится по инерции

### Замечания

- Текущее поведение: throttle=0 → PWM 1500 мкс (нейтраль). Что делает ESC — зависит
  от его прошивки (coast или лёгкое торможение).
- Торможение = отрицательный throttle. Некоторые ESC требуют «перейти через нейтраль»
  (1500 → пауза → ниже 1500) — нужно протестировать конкретный ESC.
- Можно встроить в `DriveMode` strategy pattern (уже есть после `ebb33a3`) — добавить
  метод `OnThrottleRelease()` в интерфейс.
- Для плавного торможения нужна рампа с настраиваемым slew rate.
  В kids mode такой slew уже реализован (0.3/сек) — можно переиспользовать.

**Зависимости:** тест поведения ESC.

### BUG: Ускорение при движении назад в режиме торможения

**Проблема:** при включённом режиме торможения (`braking_mode = Brake`, не накат) при движении назад машина начинает постоянно ускоряться вместо торможения.

**Ожидаемое поведение:** при движении назад (отрицательная скорость) и отпускании газа/торможении машина должна замедляться.

**Возможная причина:** логика применения `brake_slew_multiplier` или знак throttle не учитывается при движении назад. Проверить `control_loop_processor.cpp:138-140` и `kids_mode_processor.cpp:31-35`.

**Зависимости:** требуется отладка на реальном устройстве с телеметрией.

---

### BUG: Машина не двигается при малых throttle в тестовых режимах

**Проблема:** в автоматических тестовых режимах (авто-калибровка, тестовые прогоны) машина стоит на месте при малых значениях throttle.

**Ожидаемое поведение:** машина должна начинать движение при любых положительных значениях throttle.

**Возможные причины:**
1. **Мёртвая зона ESC:** ESC не реагирует на PWM < ~1550 мкс (требуется калибровка минимального газа)
2. **Неверная калибровка ESC:** не пройдена процедура калибровки мин/макс PWM
3. **Слишком низкий `kAccelerateThrottle`:** значение по умолчанию недостаточно для преодоления трения/инерции

**Что проверить:**
- Пройти калибровку ESC (мин/макс PWM)
- Увеличить `kAccelerateThrottle` в `calibration_manager.cpp`, `test_runner.cpp`
- Проверить телеметрию PWM — уходит ли сигнал на ESC

**Зависимости:** требуется отладка на реальном устройстве с телеметрией PWM и IMU.

## 5. Бинарный HTTP-эндпоинт для скачивания телеметрии

**Проблема:** скачивание телеметрии через WebSocket + JSON крайне медленное:
- JSON раздувает 80-байтовый кадр до ~300 байт (имена полей повторяются в каждом кадре)
- Пагинация по 200 кадров → 300 round-trips для 60k кадров
- cJSON сериализация float→string нагружает ESP32

**Решение:** добавить `GET /api/log.bin` — один HTTP-запрос, chunked transfer, сырые бинарные кадры.

### Протокол

```
GET /api/log.bin

Response (binary, little-endian):
  [4 bytes] uint32_t frame_count
  [4 bytes] uint32_t frame_size  (80 — для forward compatibility)
  [frame_count × 80 bytes] raw TelemetryLogFrame
```

### Выигрыш

| Метрика | JSON/WS (текущий) | Binary HTTP |
|---------|--------------------|-------------|
| Размер данных (60k кадров) | ~18 МБ | ~4.1 МБ |
| Количество запросов | 300 | 1 |
| Нагрузка на ESP32 | cJSON сериализация | memcpy |
| Оценка времени (WiFi AP) | минуты | ~7-16 сек |

### Реализация

1. HTTP handler в `http_server.cpp` — читает кадры из ring buffer, отправляет chunked response
2. JS в `app.js` — `fetch('/api/log.bin')`, парсинг `ArrayBuffer` через `DataView`, генерация CSV на клиенте

**Зависимости:** нет.

## 6. Переход на бесколлекторный (brushless) мотор

**Текущее состояние:** коллекторный (brushed) мотор со стоковым ESC от Himoto SCT-16 1:16.
Управление — стандартный RC PWM 50 Гц (1000–2000 мкс), пин GPIO14.

### Что даёт

- **Больше мощности и КПД** — меньше потерь на трение щёток, выше отдача на ватт
- **Более линейная характеристика** — dead zone 15% (`min_effective_throttle = 0.15`) частично вызвана нелинейностью коллекторного мотора; у brushless + хороший ESC зона меньше
- **Выше максимальные обороты** — больше данных для экспериментов с телеметрией на скорости
- **Долговечность** — нет износа щёток

### Риски

- Избыточная мощность для масштаба 1:16 — сложнее контролировать на низких скоростях
- Требуется перекалибровка всех параметров управления

### Что менять в прошивке

| Параметр | Изменение |
|----------|-----------|
| PWM сигнал (50 Гц, 1000–2000 мкс) | Не нужно — brushless ESC принимает тот же протокол |
| `min_effective_throttle` (0.15) | Перекалибровать — dead zone у brushless ESC другая |
| PID MotionDriver `{P=1.0, I=0.5, D=0.05}` | Перенастроить — другая динамика разгона |
| `SLEW_RATE_THROTTLE_MAX_PER_SEC` (0.5) | Возможно уменьшить — brushless реагирует быстрее |
| Инициализация ESC | Возможно добавить throttle calibration sequence (max → min → neutral) при старте |
| ZUPT/brake detection | Подстроить пороги — характеристика торможения другая |
| Аппаратная часть (GPIO, питание) | Не нужно — тот же PWM-пин GPIO14 |

### Рекомендуемое оборудование

Оптимальный вариант — **HobbyWing QuicRun 16BL30 Combo** (ESC 30A + мотор 2435SL 4500KV + программатор).

| Вариант | Ориентировочная цена |

| Комбо HobbyWing QuicRun 16BL30 | 5 000–7 000 ₽ |
| Отдельно: мотор 2435SL G3 + ESC WP-16BL30 | 5 500–8 000 ₽ |
| Доп. расходы (разъёмы, крепление) | 500–1 500 ₽ |
| **Итого** | **~5 000–9 000 ₽** |

Где искать: rc-today.ru, all4rc.ru, startrc.ru, rc-russia.ru, Ozon, AliExpress.

**Зависимости:** нет (программные изменения — только перекалибровка параметров).

### Подробный разбор плюсов и минусов каждого варианта

#### Вариант A: Апгрейд Himoto SCT-16 на brushless (5 000–9 000 ₽)

**Плюсы:**
- Минимальная стоимость — самый дешёвый путь к brushless
- Не нужно переносить электронику — ESP32, IMU, проводка уже установлены и отлажены
- Прошивка уже работает на этой платформе, протестирована
- Минимум механической работы — замена мотора + ESC, тот же крепёж GPIO14
- Сохраняется всё накопленное знание о платформе (калибровки IMU, магнитометра)
- Можно сделать за один вечер

**Минусы:**
- Масштаб 1:16 — тесно для размещения доп. датчиков в будущем
- Brushless мотор на лёгкой машине (~800 г) — рывки, сложнее контролировать на малых скоростях
- Short Course Truck — резиновые шины с грипом, не подходит для экспериментов с дрифтом/заносом
- Запчасти Himoto в Москве ограничены — если сломается шасси, ждать доставку
- Колёсная база ~170 мм — менее стабильная динамика, шумнее данные IMU из-за вибраций
- Пластиковая подвеска — больше люфтов, больше шума в акселерометре
- Потолок для развития платформы ниже — масштаб 1:16 ограничивает тюнинг

#### Вариант B: HSP Flying Fish 1 сток, brushed (8 600–11 000 ₽)

**Плюсы:**
- Готовая дрифт-платформа 1:10 из коробки — RTR, сразу можно ехать
- Масляные амортизаторы + алюминиевое шасси — меньше вибраций, чище данные IMU
- Дрифт-шины — контролируемый занос, идеально для калибровки EKF и slip angle
- Колёсная база 260 мм — стабильнее динамика, предсказуемее поведение
- Больше места в кузове для ESP32, IMU и дополнительных датчиков
- Широкая доступность запчастей в Москве (HSP — одна из самых популярных марок)
- Шестерёнчатые дифференциалы — надёжнее шариковых при нагрузках
- Можно позже апгрейдить до brushless (вариант C), когда будет нужно
- NiMH аккумулятор в комплекте — безопаснее LiPo для начала, не нужно ЗУ для LiPo

**Минусы:**
- Коллекторный мотор RC540 — dead zone, нелинейная характеристика, износ щёток
- Макс. скорость ~25–30 км/ч — может быть недостаточно для экспериментов на скорости
- NiMH аккумулятор — тяжелее и менее ёмкий, чем LiPo
- Нужно переносить всю электронику с Himoto — ESP32, IMU, проводка, крепления
- Перекалибровка всех параметров прошивки (dead zone, PID, slew rate, ZUPT)
- Два шасси в хозяйстве — Himoto станет ненужной (или запасной)

#### Вариант C: HSP Flying Fish 1 сток + ручной апгрейд на brushless (18 600–25 000 ₽)

**Плюсы:**
- Можно выбрать конкретный мотор и ESC под свои задачи (KV, ток, сенсорный/бессенсорный)
- Гибкость — например, поставить HobbyWing ESC с программируемыми параметрами
- Понимание системы — собираешь сам, лучше знаешь каждый компонент
- Все плюсы платформы HSP Flying Fish (масштаб 1:10, масляные амортизаторы, дрифт-шины)

**Минусы:**
- Самый дорогой вариант — на 3 000–5 000 ₽ дороже заводского Pro/Top (вариант D)

- Двойная работа — сначала собираешь сток, потом разбираешь и переделываешь
- Стоковые компоненты (мотор RC540, ESC, NiMH) остаются без дела

- Заводская гарантия на всю систему целиком
- Дешевле варианта C при лучшей надёжности
- Сразу готова к экспериментам — минимум времени на сборку

**Минусы:**
- Самая высокая начальная стоимость среди «разумных» вариантов (A, B, D)
- Brushless на 1:10 дрифт-машине — очень резкий отклик, на малых скоростях может быть дёрганой
- Нужно переносить электронику с Himoto (ESP32, IMU, крепления)

Ключевая идея — купить **сенсорный** (sensored) бесколлекторный мотор и подключить
его Hall-сигналы напрямую к ESP32-S3 для измерения RPM/скорости колёс.

**Плюсы:**

- HobbyWing EZRUN MAX10 G2 — программируемый, влагозащита IP64, turbo timing
- Все плюсы платформы HSP Flying Fish (1:10, масляные амортизаторы, дрифт-шины)

**Минусы:**
- Самый дорогой вариант после F — значительные вложения
- Нужно вывести Hall-сигнал из сенсорного кабеля (6-pin JST) на GPIO ESP32-S3 — пайка/разветвление

- Уровни сигнала Hall-датчиков — обычно 3.3V или 5V; если 5V, нужен делитель напряжения для ESP32 (макс. 3.3V на GPIO)

- Двойная работа: распаковал → собрал → разобрал → заменил мотор

### Техническая справка: съём данных с датчиков Холла мотора на ESP32-S3

#### Как устроены датчики Холла в сенсорном моторе

Сенсорный кабель (6-pin JST): GND, VCC (5V), Phase A, Phase B, Phase C, [опционально термистор].
Фазы A/B/C — цифровые сигналы (0V или VCC), меняются при вращении ротора.
Один электрический оборот = один цикл всех трёх фаз.
Механический RPM = электрический RPM / количество пар полюсов.

#### Расчёт скорости

```
RPM_электрический = (pulse_count × 60) / time_seconds
RPM_механический = RPM_электрический / pole_pairs

# Для EZRUN 3652SD (4 полюса = 2 пары):
RPM_мотора = RPM_электрический / 2

# Скорость колёс (HSP Flying Fish, передаточное число 1:6.052, диаметр колеса 65 мм):
speed_ms = (RPM_мотора / 6.052) × π × 0.065 / 60
```

#### Реализация на ESP32-S3

**Вариант 1: PCNT (Pulse Counter) — рекомендуемый**
- Аппаратный счётчик импульсов, не нагружает CPU
- ESP32-S3 имеет 4 PCNT-юнита, каждый с 2 каналами
- Встроенный glitch-фильтр (критично для шумных Hall-сигналов)
- Достаточно подключить одну фазу (Phase A) для RPM
- Две фазы (A + B) дают направление вращения (квадратурный режим)

**Вариант 2: GPIO-прерывания**
- Проще в реализации, но нагружает CPU на высоких оборотах
- При 20 000 RPM электрических — ~333 прерывания/сек (терпимо)

**Свободные GPIO на ESP32-S3 (из wiring_diagram.md):**
GPIO1, GPIO4, GPIO6, GPIO7, GPIO15, GPIO18, GPIO38–47 — любой подходит для PCNT.

**Уровни сигнала:**
- Если Hall-датчики питаются от 5V → сигнал 5V → нужен делитель напряжения (10K + 20K) или level shifter на вход ESP32 (макс. 3.3V)
- Если Hall-датчики толерантны к 3.3V питанию → можно запитать от 3.3V ESP32 напрямую
