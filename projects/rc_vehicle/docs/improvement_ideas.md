# Идеи для улучшения прошивки

## 1. Авто-калибровка движения вперёд без фиксированного управляющего воздействия

**Проблема:** при разном заряде батареи одно и то же PWM даёт разную скорость,
поэтому авто-калибровка (`auto_forward`) с фиксированным `throttle` ненадёжна.

**Варианты решения:**

- **(a) Ручной разгон (рекомендуется для ближайшей итерации):** оператор газует сам,
  прошивка только собирает данные IMU. Нет зависимости от точности EKF.
- **(b) PID по forward_accel:** задавать не скорость, а целевое ускорение.
  Акселерометр не дрейфует (в отличие от интеграла скорости), но нужна фильтрация вибраций.
- **(c) Speed feedback:** замкнутый контур по `ekf.speed_ms` — возможен только после
  добавления энкодера/optical flow. Без них EKF `speed_ms` считается чистым
  интегрированием акселерометра и накапливает дрейф.

**Зависимости:** вариант (a) — нет, (b) — нет, (c) — энкодер.

## 2. Тестовые автономные режимы движения

Автономные паттерны для сбора данных и настройки:

- **Круг (постоянный):** максимальный поворот + медленный газ — стабильный занос для настройки EKF/slip
- **Круг с нарастанием газа:** постепенное ускорение до появления сноса передней/задней оси — определение предела сцепления
- **Максимальное ускорение:** полный газ в течение N секунд — для калибровки `forward_accel` и EKF

### Failsafe — критичная проблема

Failsafe срабатывает через **250 мс** если нет ни RC, ни WiFi-команд.
Автономный режим будет остановлен через четверть секунды при потере WiFi.

Необходимо:
- **WiFi heartbeat из Web UI** — пока UI открыт, отправлять keepalive каждые ~100 мс
- **Кнопка аварийной остановки** в UI для немедленной отмены автономного режима
- **Таймер максимальной длительности** автономного теста (например, 30 секунд) как дополнительная защита
- **Не ослаблять failsafe** — 250 мс таймаут остаётся как safety net, heartbeat не даёт ему сработать

### Телеметрия для тестовых прогонов

Создать отдельный буфер в PSRAM для записи длинных сессий.

Текущее состояние:
- Ring-buffer: 5000 фреймов × 80 байт = **400 КБ** (при 20 Гц = ~4 мин 10 сек записи)
- PSRAM: 16 МБ доступно

Можно увеличить до 50 000 фреймов (3.6 МБ, ~41 мин записи) — малая доля PSRAM.

**Зависимости:** heartbeat + failsafe доработка.

## 3. Детский режим: ограничение по скорости/ускорению вместо PWM

**Проблема:** при разном заряде батареи одно и то же ограничение throttle PWM даёт
разную реальную скорость — режим ненадёжен.

**Текущая реализация:** multiplier-clamp (`throttle = min(throttle, limit)`, default 0.3),
slew rate 0.3/сек, anti-spin по slip angle (EKF).

### Ограничение EKF

EKF `speed_ms` без энкодеров дрейфует, а ниже 0.3 м/с возвращает 0 —
для детского режима на малых скоростях это критично.

### Реалистичный промежуточный вариант

Ограничение по **forward_accel** (IMU, не дрейфует): если ускорение превышает порог,
throttle PWM снижается. Не гарантирует точную скорость, но ограничивает динамику
разгона независимо от заряда батареи.

Полноценное ограничение по скорости — только после добавления энкодера.

**Зависимости:** ограничение по accel — нет (IMU уже есть); ограничение по speed — энкодер.

## 4. Режимы торможения

- **Плавное торможение:** при отпускании курка до нейтрали — постепенное снижение скорости (мотор-тормоз или обратный PWM)
- **Свободный накат:** при отпускании курка — не тормозить, машина катится по инерции

### Замечания

- Текущее поведение: throttle=0 → PWM 1500 мкс (нейтраль). Что делает ESC — зависит
  от его прошивки (coast или лёгкое торможение).
- Торможение = отрицательный throttle. Некоторые ESC требуют «перейти через нейтраль»
  (1500 → пауза → ниже 1500) — нужно протестировать конкретный ESC.
- Можно встроить в `DriveMode` strategy pattern (уже есть после `ebb33a3`) — добавить
  метод `OnThrottleRelease()` в интерфейс.
- Для плавного торможения нужна рампа с настраиваемым slew rate.
  В kids mode такой slew уже реализован (0.3/сек) — можно переиспользовать.

**Зависимости:** тест поведения ESC.

### BUG: Ускорение при движении назад в режиме торможения

**Проблема:** при включённом режиме торможения (`braking_mode = Brake`, не накат) при движении назад машина начинает постоянно ускоряться вместо торможения.

**Ожидаемое поведение:** при движении назад (отрицательная скорость) и отпускании газа/торможении машина должна замедляться.

**Возможная причина:** логика применения `brake_slew_multiplier` или знак throttle не учитывается при движении назад. Проверить `control_loop_processor.cpp:138-140` и `kids_mode_processor.cpp:31-35`.

**Зависимости:** требуется отладка на реальном устройстве с телеметрией.

---

### BUG: Машина не двигается при малых throttle в тестовых режимах

**Проблема:** в автоматических тестовых режимах (авто-калибровка, тестовые прогоны) машина стоит на месте при малых значениях throttle.

**Ожидаемое поведение:** машина должна начинать движение при любых положительных значениях throttle.

**Возможные причины:**
1. **Мёртвая зона ESC:** ESC не реагирует на PWM < ~1550 мкс (требуется калибровка минимального газа)
2. **Неверная калибровка ESC:** не пройдена процедура калибровки мин/макс PWM
3. **Слишком низкий `kAccelerateThrottle`:** значение по умолчанию недостаточно для преодоления трения/инерции

**Что проверить:**
- Пройти калибровку ESC (мин/макс PWM)
- Увеличить `kAccelerateThrottle` в `calibration_manager.cpp`, `test_runner.cpp`
- Проверить телеметрию PWM — уходит ли сигнал на ESC

**Зависимости:** требуется отладка на реальном устройстве с телеметрией PWM и IMU.

## 5. Бинарный HTTP-эндпоинт для скачивания телеметрии

**Проблема:** скачивание телеметрии через WebSocket + JSON крайне медленное:
- JSON раздувает 80-байтовый кадр до ~300 байт (имена полей повторяются в каждом кадре)
- Пагинация по 200 кадров → 300 round-trips для 60k кадров
- cJSON сериализация float→string нагружает ESP32

**Решение:** добавить `GET /api/log.bin` — один HTTP-запрос, chunked transfer, сырые бинарные кадры.

### Протокол

```
GET /api/log.bin

Response (binary, little-endian):
  [4 bytes] uint32_t frame_count
  [4 bytes] uint32_t frame_size  (80 — для forward compatibility)
  [frame_count × 80 bytes] raw TelemetryLogFrame
```

### Выигрыш

| Метрика | JSON/WS (текущий) | Binary HTTP |
|---------|--------------------|-------------|
| Размер данных (60k кадров) | ~18 МБ | ~4.1 МБ |
| Количество запросов | 300 | 1 |
| Нагрузка на ESP32 | cJSON сериализация | memcpy |
| Оценка времени (WiFi AP) | минуты | ~7-16 сек |

### Реализация

1. HTTP handler в `http_server.cpp` — читает кадры из ring buffer, отправляет chunked response
2. JS в `app.js` — `fetch('/api/log.bin')`, парсинг `ArrayBuffer` через `DataView`, генерация CSV на клиенте

**Зависимости:** нет.
